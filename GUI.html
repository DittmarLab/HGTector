HGTector v0.2.0 Configuration File Generator

Qiyun Zhu, Katharina Dittmar

Welcome! This page will guide you through the process of building an effective configuration file for HGTector.

Program mode:

Input data

HGTector takes one or more protein sets as input.

Each protein set should represent one genome, or a group of sequences that are expected to constitute a single genome (e.g., a bin of metagenomic contigs). Partial protein sets are okay, but it is ideal to include a large proportion of the proteins encoded by this genome, in order to fully capture the global image of the hit distribution pattern (aka. a fingerprint) of the organism.

One may provide multiple protein sets from closely-related organisms (e.g., multiple species of the same genus). In such circumstance, a consensus fingerprint describing the whole genus will be computed, and better prediction result may be achieved for individual organisms that have unique HGT activities.

Input data requirements:

Protein name: Either GenBank-style sequence titles (e.g. gi|123456789|ref|NP_123456.1| my enzyme [my species],), in which the GI and the accession number will be recognized, and the accession number (without version number) will be used as an abbreviation for the protein; or, your favored plain names (e.g., Seq001). Dots (.), pipes (|) or any confusing characters should be avoided. Anything behind the first whitespace (if any) will be intepreted as the product name (not necessary). Please make sure names are unique in each sample.

File format: Either multi-Fasta format (for example), or a plain list of sequence names (one entry per line). The latter option is only available when using BLAST, and the sequences already exist in the BLAST database.

File name: Use a short and clear file names for each protein set (e.g., MRSA.faa). The program will take the stem name (MRSA) as the sample name. Note that all file and directory names are case-sensitive.

File location: Create a folder named "input" under the working directory, and place the file(s) inside it.

There is no need to specify the samples in this GUI, but if you want to include only a subset of samples in the current analysis, or if you want to have the sample names displayed in a particular order in the report, you may enter their names below:

Example: ETEC,EAEC,EPEC

Each sample (protein set) should be given an NCBI TaxID to best describe its taxonomic position (even if it's a new organism). Please enter them below [?]:

Example: Dmel:7227,Dsim:7240,Dvir:7244

HGTector is capable of guessing the TaxIDs of in-database proteins. However this does not work for user-defined sequences. To avoid problems it is recommended to define them manually, even though you can technically leave this box blank.

Step 1: Searcher

The first step is to search a comprehensive protein database for homologs of the input proteins.

Sequence homology search has been a computationally challenging problem. It is not trivial to balance among time, precision and sensitivity. Please go through this section carefully, and choose a proper combination of tool, database and search parameters to meet the scope of your study.

Search tool

NCBI's BLAST is the current gold standard for protein sequence homology search, and is also the default tool supported in HGTector.

Being an accurate but inefficient tool, BLAST has seen limitations in handling large datasets. Faster alternatives to BLAST are worth your consideration to accelerate the analysis, such as ...

Here is an incomplete list of these tools: BLAT, Tera-BLAST, USEARCH, VSEARCH, SANS, Lambda, LAST, RAPSearch, and DIAMOND.

The current version of HGTector has the built-in support of BLAST, RAPSearch2 and DIAMOND [?]. Any tool, in or not in this list, may be used to conduct sequence similarity search, and HGTector will parse the output. See the "Pre-computed search results" section below.

The current release of HGTector has been tested on NCBI-BLAST+ 2.2.28, RAPSearch2 2.23 and DIAMOND 0.7.9.

Choice of homology search tool:   Use the NCBI BLAST server [?]

HGTector is capable of connecting to the NCBI BLAST server to perform the search. This relieves the need to deploy expensive hardware, local search programs and databases, and always guarantees the freshness of the database. All you need is a solid Internet connection. However, this has become a time inefficienct solution recently. We recommend performing local BLAST instead.

Databases

Three databases are necessary for HGTector to work: a protein sequence database, a taxonomy database, and a protein-to-taxonomy dictionary.

In the current version (0.2.0), a Python script, databaser.py, was introduced. This script automates the creation of a slim, yet comprehensive protein database based on the latest proteomic data, and the corresponding two other databases (see how).

Please have a solid Internet connection ready, move to the directory where you wish the database to be deployed, and simple execute:

python databaser.py

And wait overnight for the protein data to be downloaded, filtered and compiled into a BLAST database.

Alternatively, you may manually create / fetch customized databases.

Please enter the paths to the databases below:

Protein database: Details:

The volume of the protein database directly determines the sensitivity of HGT discovery. NCBI's nr (can be downloaded from the NCBI FTP server) is among the most comprehensive protein sequence databases, and thus maximizing the discovery of any potential gene flows (vertical or horizontal).

However, 1) nr is huge, and searching through it takes long time. Meanwhile, 2) redundant protein sequences (e.g., a few marker genes have been excessively sequenced and GenBanked) and 3) unevenly distributed taxonomic groups (e.g., there are a ton of E. coli strains AND only one giant panda sequenced) may result in biased homolog distribution pattern, thus having a deleterious effect on the HGT identification process [?].

A work-around is to check the "ignore subspecies" option (see the "Analyzer" section) to suppress redundant hits.

The built-in script databaser.py aims to solve the problems. It downloads the proteomes (*.faa) associated with the the NCBI RefSeq genomes. By default the taxon sampling is limited to microbial organisms (archaea, bacteria, fungi, protozoa and viruses), up to one representative per species, plus the NCBI-defined representative genomes. Please execute the script with "-help" for advanced usages.

Alternatively, you can create your own database with customized data. This is especially necessary when you want to focus on gene flows among a certain set of organisms. The instruction for creating customized BLAST database can be found in the official NCBI BLAST manual. Note that if you want to enable TaxID look-up, you will need to turn on advanced arguments like -parse_seqids and -taxid_map. Alternative tools have their own instructions for building databases.

Please also check out our new program SubDBer. This program subsamples a large database at designated level and generates a smaller database in which taxonomic groups are evenly (or purposely unevenly) represented. For example, an E. coli expert may use this program to extract every single strain of E. coli and Shigella, plus one representative per non-E. coli bacterial genus.

Taxonomy database: Details:

By default you should use the NCBI taxonomy database. Please download it from the NCBI FTP server, extract the contents to a local directory. Among the multiple files in the archive, all you need are two files: names.dmp and nodes.dmp. Please enter the path to the directory that contains these two files.

You may also consider building your own taxonomy database (see how) to meet special needs.

The structure of the NCBI taxonomy database (nodes.dmp and names.dmp) is straightforward. Take a look at them and you may figure out how to create your own.

HGTector only needs a small proportion of the information in these files. For nodes.dmp, only the first three columns (ID, parent ID, and rank) are necessary. For names.dmp, only the first two columns (ID and name) are necessary, plus there should be the key word "scientific name" in any of the later columns. Therefore, you can create the files like this:

nodes.dmp:
1 | 1 | no rank
123 | 1 | genus
12345 | 123 | species
13254 | 123 | species
15423 | 123 | species
...

names.dmp:
1 | root | scientific name
123 | myGenus | scientific name
12345 | mySpeciesA | scientific name
13254 | mySpeciesB | scientific name
15423 | mySpeciesC | scientific name
...

Protein-to-TaxID dictionary: Details:

The format is protein_name<tab>TaxID(s). One entry per line. If the proteins have GenBank-style titles, you should use accession numbers or GIs instead of full names. In cases one protein sequence is shared by multiple organisms, you may include multiple TaxIDs separated by comma.

The most comprehensive dictionary can be downloaded from the NCBI FTP server. It contains all NCBI protein entries. However, please be aware that it will consume dozens of GB of memory when loaded.

Official BLAST databases (such as nr) have built-in TaxIDs, and HGTector is capable of looking them up. You may also compile TaxIDs into your customized BLAST database using the -taxid_map argument. However, the query process is computationally inefficient. We recommend having a standalone dictionary anyway.

If you already have a BLAST database that has built-in TaxIDs, you can do this to extract them into a standalone dictionary:

blastdbcmd -db mydb -entry all -outfmt "%a %T" > mydict.txt

Note: Query proteins do not have to be in the dictionary, as HGTector will refer to the taxonomy of the parent protein set.

Hits without identifiable taxonomy will be discarded.

Pre-computed search results

Instead of having HGTector run homology searches, you may provide it with pre-computed search results.

Any search tool, as long as its output can be converted into the standard BLAST tabular format (-outfmt 6 or 7) (alternatively...), can be chosen to work with HGTector. Many search tools have options to print results in this format. For example, you can do -out=blast8 for BLAT, or -blast6out for USEARCH.

Actually, HGTector only considers field 1 (query id), 2 (subject id), 3 (percentage identity), 11 (E-value) and 12 (bit score). It is okay if the rest fields are filled with unrelated information. You may optionally attach target sequence (sseq) in an extra field (13).

Important: HGTector needs to know how well each query protein aligns to itself, in order to normalize bit scores among proteins. Therefore, it is ideal that the pre-computed search results already contain this information. If not ...

If not (e.g., search a user-defined protein sequence against a standard database), HGTector will call the corresponding search program (if supported) to perform self-alignment for these proteins. Otherwise, HGTector will take the top hit of each protein as the estimate of the self-alignment result (not accurate).

Keep one result file per protein set. Name them using the same stem file names as the protein sets (e.g., the input sequence files are HIV.faa and HPV.faa, then the corresponding pre-computed search result files may be HIV.tab and HPV.tab).

Taxon grouping scenario

All taxa are divided into three groups: self, close and distal, based on their evolutionary distance to the input genomes. Typically, the self group should contain all input genomes, the close group should contain the close relative lineages of the self group, and the distal group constitutes everything else.

A homology match between a distal group organism and a self group one is considered a potential horizontal gene flow, while that between close and self is more likely but vertical inheritance within the lineage.

It is important to define a proper grouping scenario to match the scope of your study (Are you interested in HGTs on what taxonomic level?). For examples ...

In our original study we were interested in the acquired genetic materials in seven genomes belonging to the Rickettsia spotted fever group (SFG). So we defined self = SFG (TaxID: 114277), close = order Rickettsiales (TaxID: 766) (self organisms are automatically excluded from the close group) and distal = everything else (no need to define).

Similarly, when we studied HGTs in Bartonella, we defined self = genus Bartonella, close = order Rhizobiales.

If you are interested in bacteria-to-human HGTs (not very likely), you may define self = great apes, and close = eukaryotes.

If you are studying gene exchanges among E. coli strains, you will have to have an E. coli (vertical) phylogenetic tree in mind, and define branches that are close to the organism of interest as close, while others as distal. But please be prepared to experience a lower predicting power on very close HGTs.

Hit pattern partitioning

For each of the self, close and distal groups, once the weights of all protein-coding genes are calculated, HGTector will cluster the weight population, and attempt to separate the "atypical" part (which has very low weights) from the "typical" part using a cutoff.

Several statistical approaches are implemented in HGTector to determine whether a weight population can be comfortably partitioned, and what would be a proper cutoff for it.

Exclude outliers at high end, using the method. [?]

These outliers are genes with unexpected high number of hits. They may be caused by database redundancy. Excluding them prior to clustering may help to suppress false results.

Test for non-unimodality (requires R package diptest). [?] P-value threshold: [?]

Perform Hartigan's dip test using R package diptest to assess the significance of non-unimodality of weight distribution. The smaller the p-value is, the more comfortable it is to split the population into two or more clusters.

If p-value >= this threshold (meaning the non-unimodality is insignificant), the program will consider that there is inadequate statistical support for dividing the population into two clusters. In such circumstance, the global cutoff will be used.

Perform kernel density estimation using: [?]

Kernel density estimation (KDE) is a statistical approach to estimate the probability density function of a weight population. It can be considered as a smoothed histogram. The program will perform KDE using R command density with its default parameters (Gaussian kernel), or using built-in Perl code with parameters that mimic the R command.

Bandwidth selection factor: [?] Plot kernel density function on screen.

A bandwidth indicates how "smooth" the density function curve is. The program uses Silverman's rule-of-thumb equation to compute an optimal bandwidth. One can further apply a factor on it. The smaller the factor is, the more resolved (in terms of peaks/pits) the curve will be.

Hit distribution scoring

For each protein-coding gene, three weights are computed for the self, close and distal groups. By default, a weight is defined as the sum of normalized bit scores of hits belonging to each group. A fingerprint is the distribution pattern of the three weights of all protein-coding genes within one or more genomes.

One global fingerprint for all protein sets. (Alternatively, each set has its own fingerprint.)

Use weight (sum of scores) instead of number of hits (recommended).

Normalize bit scores as subject divided by query (recommended).

3D plotting method

Bidirectional Best Hit (BBH) is the traditional approach for identifying gene flows. For a protein-coding gene, if the best non-self match is distal, instead of close, then this gene is predicted as HGT-derived.

This conventional method is sensitive to stochastic events that lead to pseudo positives and negatives. It is therefore depreciated. HGTector contains this option for comparison purpose only.

Use the conventional best BLAST match method.

Note: if BBH is chosen, the program will perform an extra reversal BLAST step on every HGT candidate, which takes some time.
